# Tool Schema Appendix — loreholm

This document defines the **canonical MCP tool schemas** for interacting with loreholm.

These tools are the **only supported interface** for LLMs to read from or write to memory.

If it is not expressible through these tools, it does not exist.

---

## Design Goals

All tools must be:

- **Explicit** — no hidden behavior
- **Deterministic** — same input → same effect
- **Inspectable** — results can be audited by humans
- **Composable** — tools do one thing well
- **Safe to retry** — idempotent where possible

---

## Common Conventions

### Identifiers
- All IDs are opaque strings generated by the system
- LLMs must treat IDs as stable references

### Timestamps
- ISO 8601 strings (UTC)

### Confidence
- Float in range `[0.0, 1.0]`
- Represents *confidence in correctness*, not importance

### Tags
Common tags include (not exhaustive):
- `user_stated`
- `inferred`
- `speculative`
- `todo`
- `question`

### Per-database graph schema

Entity types, relationship types, and aliases are authored **per database**
in the local dashboard. The authoritative graph schema lives on the local
side; the cloud API caches it in the `schema_json` column on
`database_targets` and uses the `profile_hash` column as the staleness
signal for the whole target row. See `docs/11_ApiKeyAuth.md` for the
storage columns and `docs/07_BYODB.md` §7 for the synchronization flow.

All API keys pointing at the same database share the same graph schema,
because they write into the same ArcadeDB instance. Different *capability*
exposure across keys (read-only vs. read-write, tool filtering) is handled
separately via the per-target `tool_manifest_json` overlay, not by forking
the schema. At MCP connection time the server composes the final tool
surface from the base MCP tool set, the per-target tool manifest, and the
per-database graph schema.

### Entity type validation (write-strict, read-loose)

Entity `type` is validated **only on the write path**. `EntityInput` (and the
upsert path in `arcadedb_store.upsert_entities`) runs every incoming `type`
through `canonicalize_entity_type`, rejecting anything not in the currently
allowed set for the database.

The read path is deliberately loose: `EntityRef`, `ContextEntity`, and
`TopEntity` all expose `type` as a plain `str` with no validator. Search,
context, and stats responses will serialize whatever label a node carries in
ArcadeDB, even if that label is no longer part of the authored schema.

**This asymmetry is intentional and load-bearing for schema evolution.**
In the multi-schema model each database owns its own entity/relationship
vocabulary and users can edit it over time. Keeping reads loose means:

- Removing an entity type from the schema requires **no data migration**.
  Stale nodes still surface in search/context results; they just drop out of
  the write-side tool enum so no new ones accumulate.
- Renames via soft alias work automatically on the read side (the write side
  canonicalizes the alias to the new label; reads pass through either form).
- Occasional stale-type labels do not cause response-level serialization
  errors that would be opaque to end users.

Strict read validation would force a data-migration layer for every schema
edit, breaking the additive-and-forgiving evolution model.

Do **not** introduce read-path entity-type validation later as a "hardening"
pass without revisiting this decision. The correct place to surface stale or
orphaned labels is the local dashboard's schema view, which diffs the
authored schema against labels actually present in the graph — not runtime
response validation. Downstream code that branches on entity type must handle
unknown types defensively, since the read validator will not catch them.

### Entity type renames (soft alias)

Renaming an entity type in a database's schema is handled as a **soft alias**,
not a data migration.

When a user renames `Old` → `New` in the local dashboard schema editor, the
schema record stores an alias mapping `Old → New` alongside the new
authoritative label. The behavior is:

- **Write path:** `canonicalize_entity_type` resolves both `Old` and `New` to
  the current authoritative label (`New`). New upserts land with the new
  label; existing nodes keep whatever label they were created with. The
  tool-side enum advertises only `New`, so the LLM naturally uses the new
  form.
- **Read path:** already loose (see above). `Old`-labeled nodes continue to
  surface in search, context, and stats results with their original label.
  The LLM sees a mix of `Old` and `New` and tolerates it.
- **No data migration.** Historical nodes are never rewritten. The graph is
  allowed to carry both labels indefinitely.

Aliases are **cumulative, not chained.** Renaming `A` → `B` then `B` → `C`
must produce an alias map where both `A` and `B` resolve directly to `C`, not
`A → B → C`. The resolver should never walk a chain.

An alias is permanent once recorded — removing an alias from the schema
would mean rejecting writes of the old form while old-form data still sits
in the graph, which defeats the purpose. The dashboard UI should treat alias
entries as append-only unless the user explicitly cleans up stale graph data
first.

Aliases are a property of the **per-database schema** (not the per-key tool
manifest), because they describe the shape of the data, not the capabilities
exposed to a specific client.

## Write Tools

### 1. `loreholm_upsert_entities`

Create or retrieve canonical entities.

#### Purpose
Ensure entities are deduplicated and consistently referenced.

#### Input Schema
```json
{
  "entities": [
    {
      "name": "string",
      "type": "Person | Project | Tool | Concept | Place | Other (case-insensitive input)",
      "aliases": ["string"],
      "merge_into": "string (optional Entity.id from loreholm_search_similar_entities)"
    }
  ]
}
````

#### Output Schema

```json
{
  "entities": [
    {
      "entity_id": "string",
      "name": "string",
      "type": "string",
      "aliases": ["string"],
      "created": true
    }
  ]
}
```

#### Notes

* Matching is name + alias based (POC-level)
* `created=false` indicates an existing entity
* No silent merging across different types
* Entity `type` is normalized to canonical title case on write (`person` -> `Person`)
* `merge_into` is an optional dedup hint pointing at an existing `Entity.id`
  (typically obtained from `loreholm_search_similar_entities`). The reconciler
  honors the hint when the proposal's embedding is within the per-database
  review threshold of that target; otherwise the row downgrades to
  `needs_review` so the disagreement can be inspected. A stale or unresolved
  id falls through to the standard vector-nearest decision.

### 2. `loreholm_write_memory`

Write an atomic, human-readable memory.

#### Purpose

Persist a fact, observation, or note derived from a conversation.

#### Input Schema

```json
{
  "text": "string",
  "about_entity_ids": ["string"],
  "confidence": 0.0,
  "tags": ["string"],
  "source_ref": {
    "conversation_id": "string",
    "message_ids": ["string"],
    "platform": "string (optional)",
    "started_at": "ISO-8601 string (optional)",
    "messages": [
      {
        "id": "string",
        "role": "string (optional)",
        "text": "string (optional)",
        "timestamp": "ISO-8601 string (optional)"
      }
    ]
  }
}
```

#### Output Schema

```json
{
  "memory_id": "string",
  "timestamp": "ISO-8601 string",
  "linked_entities": ["string"]
}
```

#### Rules

* Max **5 memories per interaction**
* `text` must be human-readable and standalone
* `source_ref` is mandatory
* Prefer deterministic `conversation_id` and `message_ids` from the host thread/message IDs
* Include `source_ref.messages[*]` metadata when available to avoid `unknown`/empty Message node fields
* Low confidence is preferred over false certainty

---

### 3. `loreholm_link_entities`

Create an explicit relationship between entities.

#### Purpose

Capture relationships that are meaningful beyond a single memory.

#### Input Schema

```json
{
  "from_entity_id": "string",
  "to_entity_id": "string",
  "relationship": "string",
  "confidence": 0.0,
  "reason": "string"
}
```

#### Output Schema

```json
{
  "edge_id": "string",
  "from_entity_id": "string",
  "to_entity_id": "string"
}
```

#### Notes

* Relationships are directional
* Free-text `relationship` is allowed at POC stage
* `reason` must justify the link

---

### `loreholm_delete_entities`

Delete entities by ID.

#### Purpose

Remove bad or obsolete entities from the graph.

#### Input Schema

```json
{
  "entity_ids": ["string"]
}
```

#### Output Schema

```json
{
  "deleted_entity_ids": ["string"],
  "not_found_entity_ids": ["string"],
  "deleted_count": 0
}
```

#### Notes

* Deletion uses `DETACH DELETE` semantics
* Unknown IDs are reported in `not_found_entity_ids`
* Operation is safe to retry

## Read Tools

### 4. `loreholm_search`

Search memories and entities by relevance.

#### Purpose

Retrieve candidate context for grounding a conversation.

#### Input Schema

```json
{
  "query": "string",
  "top_k": 10,
  "filters": {
    "entity_types": ["string"],
    "tags": ["string"],
    "since": "ISO-8601 string"
  }
}
```

#### Output Schema

```json
{
  "items": [
    {
      "memory_id": "string",
      "text": "string",
      "confidence": 0.0,
      "timestamp": "ISO-8601 string",
      "entity_refs": [
        {
          "entity_id": "string",
          "name": "string",
          "type": "string"
        }
      ],
      "source_ref": {
        "conversation_id": "string"
      }
    }
  ],
  "suggested_context_block": "string",
  "retrieval_mode": "vector|hybrid|lexical_fallback",
  "diagnostics": {
    "vector_ok": true,
    "fallback_reason": "string|null",
    "vector_candidates": 0,
    "lexical_candidates": 0,
    "final_count": 0,
    "query_terms": ["string"]
  }
}
```

#### Notes

* Vector-first search with lexical fallback for long/natural-language prompts
* Response includes retrieval diagnostics so clients can detect fallback behavior
* Results remain explainable and inspectable

---

### 4b. `loreholm_search_similar_entities`

Find existing entities that may already represent a proposed name.

#### Purpose

Let the calling LLM dedupe entities at write time. Call before
`loreholm_upsert_entities` whenever the proposal might already exist; pass
the returned `entity_id` back as `merge_into` to declare the equivalence.

#### Input Schema

```json
{
  "query": "string",
  "top_k": 5,
  "type": "string (optional — Entity type filter)"
}
```

#### Output Schema

```json
{
  "matches": [
    {
      "entity_id": "string",
      "name": "string",
      "type": "string",
      "aliases": ["string"],
      "similarity": 0.0,
      "lexical_score": 0.0,
      "sources": ["vector", "lexical"]
    }
  ],
  "message": "string",
  "reconciler_lag_seconds": 0.0
}
```

#### Notes

* Hybrid scoring: vector-nearest (cosine via
  `vectorNeighbors('Entity[embedding]')`) union lexical (tokenized
  `name_norm` / `aliases_norm` match). Lexical-only candidates carry a flat
  penalty.
* Read-only — does not stage anything.
* `type` is optional. Reconciler-side dedup scopes by type, but the search
  tool intentionally does not, because the LLM may be probing before it
  knows the type.

### 5. `loreholm_context`

Expand context around known entities.

#### Purpose

Assemble a compact context pack via graph traversal.

#### Input Schema

```json
{
  "entity_ids": ["string"],
  "depth": 1,
  "limit": 20
}
```

#### Output Schema

```json
{
  "memories": [
    {
      "memory_id": "string",
      "text": "string",
      "confidence": 0.0,
      "entity_refs": ["string"]
    }
  ],
  "entities": [
    {
      "entity_id": "string",
      "name": "string",
      "type": "string"
    }
  ],
  "suggested_context_block": "string"
}
```

#### Rules

* Default `depth ≤ 2`
* Favor relevance over completeness
* Context block must be concise

## Error Handling (All Tools)

Errors must be:

* Explicit
* Structured
* Non-destructive

Example:

```json
{
  "error": {
    "code": "INVALID_ENTITY_ID",
    "message": "Entity ID not found",
    "details": {
      "entity_id": "string"
    }
  }
}
```

---

## Final Rules for Agents

Agents MUST:

* Use these tools as specified
* Prefer fewer, higher-quality memories
* Preserve provenance and confidence
* Avoid silent mutation

Agents MUST NOT:

* Invent tool behavior
* Write memory implicitly
* Overload tools with hidden semantics

If a memory cannot be justified later, it does not belong in loreholm.
