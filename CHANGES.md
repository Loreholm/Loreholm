- **ArcadeDB and Bifrost moved off the tailscale netns onto the default compose bridge — defense-in-depth so 2480/8080 are no longer bound to the user's Tailnet interface.** With `local-dashboard` already on the LAN bridge (the 2026-01 hardening that pulled 4466 off Tailnet), there was no remaining reason for `loreholm-arcadedb` or `loreholm-bifrost-proxy` to share `loreholm-tailscale`'s netns — the dashboard reaches them by container DNS now, not loopback. But because both still ran with `network_mode: service:tailscale` and bound to `0.0.0.0` (ArcadeDB's `arcadedb.server.httpIncoming.host` isn't set; Bifrost's default), their listening sockets were live on the user's `tailscale0` interface. The CHANGES.md entries from the past few weeks (the `acl_policy_path` deprecation silently disabling the policy; the `autogroup:member` Policy v2 mismatch crash-looping the ACL loader) are exactly the failure mode this leaves us exposed to. Dropped `network_mode: service:tailscale` and `depends_on: tailscale (service_healthy)` from both services in the install/update compose generators (`web/install.sh`, `web/update.sh`, `web/install.ps1`, `web/update.ps1`) and the onboarding compose template (`api/app/onboarding/router.py`); the containers now live on the default project bridge alongside `loreholm-local-dashboard`. Switched the local-dashboard env vars from `LOCAL_DASHBOARD_ARCADEDB_HOST=loreholm-tailscale` / `LOCAL_DASHBOARD_BIFROST_URL=http://loreholm-tailscale:8080` to `loreholm-arcadedb` / `http://loreholm-bifrost-proxy:8080` (direct Docker DNS into the same bridge). Updated the matching defaults in `api/app/local_dashboard/core/config.py` so a stale install without env vars still resolves; the legacy alias check in `db/registry.py:_is_local_host_alias` already treats `loreholm-tailscale` as a local-host alias, so persisted `databases.json` records pointing at the old name continue to resolve through `_resolve_arcadedb_host`. After this change the only thing in the tailscale netns is the `loreholm-local-dashboard-endpoint` shim on `:8081` — which matches the comment block at the shim's compose entry ("the only thing on this host with a Tailnet IP"). Users pick up the change by re-running `update.sh` / `update.ps1`. Rather than a background poll loop keeping the cloud cache fresh, the cloud now routes every query through the local dashboard's new `POST /api/sync/query` endpoint, and the response envelope carries the current `profile_hash` alongside the rows.
